# v2.1.0 Single-Record Workflow Tools Implementation Plan

> **For agentic workers:** implement this plan task-by-task. Keep checkboxes current as work lands. Do not skip the transaction and validation tasks; they are what make the workflow safe on a single canonical `out/<slug>/`.

## Product Goal

V2.0 changed storage: one current protocol record lives at `out/<slug>/`, and history lives in the nested `out/.git` repo.

V2.1 turns that storage model into a usable workflow:

1. Inspect the current record without opening JSON by hand.
2. Translate the current record without re-crawling.
3. Search/analyze one field and optionally apply a validated change.
4. Manually edit one field with schema validation.
5. Browse history, diff commits, and restore a previous valid version.

The core user story is:

```bash
bash run.sh get pendle fundingRounds[0]
bash run.sh analyze pendle fundingRounds --query "verify latest Pendle funding rounds"
bash run.sh analyze pendle fundingRounds --query "verify latest Pendle funding rounds" --apply
bash run.sh history pendle
bash run.sh diff pendle HEAD~1 HEAD
bash run.sh restore pendle HEAD~1
```

## Non-Goals

- No multiple protocol versions shown side-by-side in the browser. The browser shows one current protocol record; history/diff are git-backed.
- No slash-command surface in v2.1. Users invoke via `bash run.sh <command> ...`.
- No interactive TUI or REPL.
- No automatic migration of legacy `out/<runId>/<slug>/` directories.
- No unsafe write path. Every write command validates, commits, and rolls back on failure.

## Command Surface

V2.1 ships these workflow commands:

| Command | Writes? | Purpose |
|---|---:|---|
| `get <slug> <jsonpath>` | No | Print one value from `record.json`. |
| `set <slug> <jsonpath> <json-value>` | Yes | Manual field edit with schema validation and commit. |
| `analyze <slug> <jsonpath> --query <text> [--apply]` | Optional | Search/analyze one field. Default is proposal-only; `--apply` writes the proposed field value. |
| `refresh <slug> <subtask>` | Yes | Re-run a broad subtask (`metadata`, `team`, `funding`, `audits`) and merge through `mergeR2`. |
| `i18n <slug> [--locales ...]` | Yes | Re-translate current record, then run post-processing. |
| `history <slug> [--limit N]` | No | Show git history for one protocol. |
| `diff <slug> [from] [to]` | No | Show git diff for one protocol. Defaults `HEAD~1..HEAD`. |
| `restore <slug> <sha>` | Yes | Restore a previous commit, revalidate, commit the restore. |

`analyze` is intentionally separate from `refresh`:

- `refresh` is subtask-level and useful when a whole area is stale.
- `analyze` is field-level and matches the product goal of "search/analyze/modify a key".

## Acceptance Scenarios

- **Single current version:** after multiple crawls and edits, the browser still displays `out/pendle/record.json` as the current Pendle record, not a list of run-id copies.
- **Translate in place:** `bash run.sh i18n pendle --locales zh_CN` creates or updates `record.full.json`, `record.import.json`, and `meta.json.i18n`, then commits exactly one logical `i18n(pendle): zh_CN` change.
- **Manual key edit:** `bash run.sh set pendle description '"new text"'` writes only after full-record validation passes; failed validation leaves file content and git status unchanged.
- **Field research:** `bash run.sh analyze pendle fundingRounds --query "verify funding"` prints a proposed value plus evidence without writing by default.
- **Field research apply:** the same command with `--apply` validates the proposed record, writes the field, updates sidecars if needed, commits `analyze(pendle) fundingRounds`, and refreshes `index.html`.
- **Broad refresh:** `bash run.sh refresh pendle funding` loads existing `record/findings/changes/gaps`, wraps both old and new state as merge envelopes, calls `mergeR2`, writes `merged.record` plus sidecars, validates, post-processes, commits.
- **Restore safety:** if `restore` checks out a record that fails the current schema, it rolls back to `HEAD`, exits non-zero, and leaves the slug clean.

## Architecture

### Flow

```text
run.sh <command> ...
  -> framework/cli.mjs dispatcher
  -> framework/commands/<command>.mjs
  -> read current out/<slug> state
  -> command-specific work
  -> validate full record when writing
  -> write canonical artifacts
  -> git commit one logical user action
  -> rebuild out/index.html for write commands
```

### Shared Safety Rule

Every write command must behave transactionally for the slug:

```text
preflight: ensure out repo + ensure slug clean unless --force-overwrite
do work in memory or temp/debug files
validate candidate canonical record
write canonical files
run post-processing when canonical import/full/meta can be affected
commit explicit paths under <slug>/
on any failure after checkout/write: resetSlugToHead(out, { slug })
```

Ignored `_debug/` files may remain for triage. Tracked canonical files must return to `HEAD` on failure.

### Files

| Path | Action | Responsibility |
|---|---|---|
| `framework/cli.mjs` | Modify | Add dispatcher for workflow commands. Unknown command falls through to existing crawl behavior. Help text documents workflow commands. |
| `framework/schema-validator.mjs` | Modify | Add main guard and export `validateRecord(record, manifestOrSchema)` / `validateFile(file, schema)` while preserving CLI exit behavior. |
| `framework/jsonpath.mjs` | Create | Tiny JSONPath dialect for `a.b[0].c`: parse, get, set. No filters, no recursive descent, no implicit array growth. |
| `framework/record-state.mjs` | Create | Load/write `{ record, findings, changes, gaps }` envelopes for a slug. Used by `refresh` and `analyze`. |
| `framework/slug-transaction.mjs` | Create | Shared helpers for clean preflight, rollback, commit, and browser rebuild after write commands. |
| `framework/refresh-runner.mjs` | Create | Refresh-specific wrapper around current R1 subtask flow. It renders the real subtask prompt and calls `runSubtask` with schema slices and beta sidecar schemas. |
| `framework/key-analyzer.mjs` | Create | Field-level research runner. Takes `{ slug, jsonpath, query, currentValue, record, evidence }`, runs Claude with a strict patch schema, optionally routes approved search requests through existing search channels, and returns a proposal. |
| `framework/commands/*.mjs` | Create | One module per command: `get`, `set`, `analyze`, `refresh`, `i18n`, `history`, `diff`, `restore`. |
| `tests/framework/...` | Create/Modify | Unit and command tests described below. |
| `README.md`, `README.zh-CN.md`, `commands/protocol-info.md`, `CHANGELOG.md` | Modify | Document workflow commands and v2.1 release. |

## Data Contracts

### Record Envelope

`record-state.mjs` standardizes the shape passed into merge/analyze code:

```js
{
  record: object,
  findings: array,
  changes: array,
  gaps: array
}
```

Missing sidecars load as empty arrays. `mergeR2` must only receive this envelope shape; never pass a bare record or raw subtask result.

### Analyze Proposal

`key-analyzer.mjs` returns:

```js
{
  ok: true,
  path: "fundingRounds",
  current_value: unknown,
  proposed_value: unknown,
  confidence: 0.0,
  rationale: "short explanation",
  findings: [],
  changes: [],
  gaps: []
}
```

If it cannot support a change, it returns `ok:false` with `reason` and optional gaps. `analyze` without `--apply` prints this proposal and writes no canonical files.

### Post-Processing Rule

Run `framework/cli/post.mjs` after commands that can change import/full/meta outputs:

- `crawl` already does this in v2.0.
- `i18n` always runs post after the i18n stage.
- `set --no-post` is not supported in v2.1; run post after successful set to keep `record.import.json` aligned.
- `analyze --apply`, `refresh`, and `restore` run post after writing the candidate record.

## Release Criteria

V2.1 is done when:

- A user can inspect, translate, manually edit, analyze/apply a field, refresh a subtask, diff, and restore a protocol without creating a second displayed protocol version.
- Every write command leaves either one clean commit or no canonical change.
- `out/index.html` reflects the current canonical record after write commands.
- Full tests pass.
- README/command docs explain the single-record workflow clearly.
